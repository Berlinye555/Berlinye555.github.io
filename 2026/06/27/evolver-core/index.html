Evolver：GEP 基因编程自进化引擎

Evolver：GEP 基因编程自进化引擎

来源：agent-frameworks/evolver/，论文 arXiv 2604.15097
团队：EvoMap
语言：JavaScript/Node.js，有自己的 EvoMap Hub 生态

一句话总结

用**基因（Gene）+ 胶囊（Capsule）**替代 SKILL.md 文件作为进化的载体——Gene 定义”该做什么改进”，Capsule 记录”改进后发生了什么”。加上变异（mutation）、表观遗传（epigenetics）、固化（solidify）等生物学隐喻，构建了一套完整的基因编程进化系统。

核心创新：用生物学隐喻替代文件系统

所有其他项目都把”技能/知识”存为文件：

项目 存储方式
EvoSkill .claude/skills/<name>/SKILL.md
self-improving-skills ~/.claude/skills/<name>/SKILL.md
SkillRL memory_data/<task>/claude_style_skills.json
Evolver Gene + Capsule 对象（内存 + 磁盘 + Hub 同步）

Evolver 不写技能文件——它创建基因和胶囊。

Gene（基因）结构

对比 SKILL.md:

  • SKILL.md: 非结构化的 markdown 文本，靠 LLM 理解
  • Gene: 严格结构化的 JSON 对象，程序化验证和路由

Capsule（胶囊）结构

Gene = 假设（”我们应该这样改进”），Capsule = 实验（”我们试了，结果是…”）。

这和 EvoSkill Feedback Descent 的 Proposer → Evaluator → 保留/丢弃 完全对应：

  • Gene ≈ Proposer 的输出（候选改进）
  • Capsule ≈ Evaluator 的结果（带证据的判断）

进化管线

表观遗传（Epigenetics）

Evolver 引入了一个其他项目都没有的概念：表观遗传标记。

  • 基因不变（Gene 的 strategy/validation 不改），但可以被”标记”为抑制/激活
  • 类似于生物的 DNA 甲基化——基因序列没变，但表达变了
  • 这解决了其他系统的一个问题：一条技能偶尔不适用 vs 永远不适用，如何区分？用标记而非删除

和 self-improving-skills 的对比

Evolver self-improving-skills
进化载体 Gene + Capsule（结构化 JSON） SKILL.md（非结构化 markdown）
验证方式 沙箱执行 + Capsule 评分 格式校验 + 人工判断
状态管理 epigenetic_marks（抑制/激活） active → stale → archived
知识共享 EvoMap Hub（A2A 广播） Team repo（origin-hash sync）
安全 沙箱执行 + constraints 备份 + 回滚
生态 有 Hub 市场、自动任务撮合（ATP） CLI 插件、手动命令
成熟度 论文支撑，商业生态 个人项目，日用可靠

最有价值的概念：Gene + Capsule 分离

把”改进建议”和”改进结果”分开是两个正交的数据结构，这是 Evolver 最值得学走的设计。

在 self-improving-skills 里，两者是混在一起的——SKILL.md 既是”建议”也是”经过验证的知识”。如果能引入 Gene/Capsule 分离：

  • Gene = distiller 提出的候选改进
  • Capsule = 使用该技能后的效果记录（从 usage_store 的 use/view/patch 统计推导）
  • Epigenetic marks = 根据 Capsule 的 outcome 自动调整 Gene 的激活状态

一句话带走

不用 markdown 文件，用结构化的基因（Gene）和胶囊（Capsule）来承载 Agent 的进化知识。 Gene 是假设，Capsule 是实验证据。表观遗传标记（epigenetics）让基因可以在不删除的情况下被抑制或激活。这是一套比 SKILL.md 更严谨、更可程序化操作的进化载体设计。
