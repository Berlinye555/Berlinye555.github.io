self-improving-skills：日常工作的经验蒸馏引擎

来源：Claude Code 插件 self-improving-skills@claude-self-improving-skills v0.9.2
作者：UniM0cha，灵感来自 Nous Research Hermes Agent
安装时间：2026-06-26

一句话总结

在 Claude Code 的 Stop hook 中检测复杂工作 → 弹窗提醒蒸馏 → 子 agent 读 transcript 写 SKILL.md → curator 定期清理 stale 技能。全链路 fail-safe，任何异常都不阻塞会话。

整体架构

四个 Hook 的职责边界

1. Stop Hook — 复杂度检测（核心）

文件：hooks/distill-nudge.sh → scripts/analyze_turn.py（409行）

关键设计决策（从 dev-log hook 的 396 个 transcript 零触发失败中总结）：

问题 dev-log 的做法 analyze_turn.py 的修正
检测 tool call grep "tool":"Edit" 解析 assistant row 的 message.content[].type == "tool_use"
检测已蒸馏 搜索 “distill” 字符串 检查是否调用了 subagent_type 含 skill-distiller 或写入了 SKILL.md
输出方式 stderr + exit 2 stdout JSON + exit 0
重复提醒 无防护 记录 nudge 时的 row count，从该位置重新计数

2. PreToolUse Hook — 写前备份

文件：hooks/backup-skill.sh

只在目标是 ~/.claude/skills/**/SKILL.md 时触发，复制到 ~/.claude/self-improve/skill_backups/<name>.bak。PostToolUse 校验失败时用于回滚。

3. PostToolUse Hook — 写后校验

文件：hooks/validate-skill.sh → scripts/validate_skill.py（321行）

校验规则：

  • 必须以 --- frontmatter 开头并闭合
  • name 必填，≤64 char，仅小写/数字/连字符
  • description 必填，≤1024 char（>500 软警告）
  • body 非空
  • 全文件 ≤100000 char
  • 如果格式错误 + 有备份 → 自动回滚到编辑前版本（事务安全性）
  • 如果格式错误 + 无备份（新文件）→ 只警告，不回滚

校验通过后自动打 metadata.provenance: self-improving-skills 标记，让 curator 能区分 agent 蒸馏的技能和用户手写的技能。

4. SessionStart Hook — 会话初始化

文件：hooks/session-init.sh → scripts/session_init.py（239行）

注入三条信息到会话上下文：

  1. 自改进循环已激活 + 如何使用 /distill-skill
  2. 当前已学习的技能数量
  3. curator 触发判断（首次安装 → 种子时钟不执行；7天+ → 运行自动清理）

Distiller 子 Agent 的决策树

文件：agents/skill-distiller.md（145行）

反模式检测（明确不捕获的内容）：

  • 一次性任务叙事（”2026-06-03 修了构建”）
  • 环境相关 workaround
  • 负面工具声明（”工具 Z 不行”——很快就过时）
  • 文档已有的显而易见知识

Usage Store — 技能状态机

文件：scripts/usage_store.py（320行）

状态转换逻辑（curator_transitions.py）：

为什么 patch 计数在 PostToolUse 而非 Stop？ 因为 distiller 是后台子 agent，它的 transcript 是独立的——Stop scanner 读不到。PostToolUse hook 在子 agent 内部也会触发，所以不会漏。

关键设计原则

Fail-Safe 是最高优先级

每个 hook 的入口都是 try...except: approve/silent。analyze_turn.py 有 4 层保护：

  1. JSON 解析失败 → approve
  2. transcript 文件不存在 → approve
  3. stop_hook_active 循环守卫 → approve
  4. 任何未预期异常 → approve

一个损坏的分析器永远不能卡住会话。

Nudge Once Per Segment

记录 nudge 时的 transcript row count，下次只从该位置之后计数。这意味着：

  • 用户拒绝了蒸馏（”这次是一次性工作”） → 同一个 segment 不会再次提醒
  • 只有新的工作积累超过阈值才会再次触发

维护操作不污染统计

curation 期间的 Skill/Read 调用不计入 usage 统计——否则每次 /curate-skills 都会把 stale 技能的 idle clock 重置，时间淘汰机制失效。

和 EvoSkill Feedback Descent 的对比

Feedback Descent self-improving-skills
优化目标 Agent Program（prompt + skills 集合） 单个 SKILL.md 文件
Proposer LLM 分析 benchmark 失败 LLM 读 transcript
Evaluator val 集量化跑分 用户主观判断
种群管理 Frontier top 3 无（单技能线性演化）
触发 算法循环自动 Stop hook 弹窗
维护 无 curator active→stale→archived 状态机
安全性 无特殊机制 全链路 fail-safe + 备份回滚
代码量 133行核心算法 + 712行 loop 409行分析器 + 321行验证器 + 320行存储

一句话带走

在正确的时机（Stop hook），用正确的信息（transcript JSONL），做正确的决策（阈值 + nudge-once 守卫），以正确的方式写技能（patch 优先，创建是最后手段），并持续维护（curator 状态机）。 全链路 fail-safe 的设计让它在 396 个 transcript 的实战中一次也没有错误阻塞会话。
