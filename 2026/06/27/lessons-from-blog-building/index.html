一天搭建博客踩过的坑——用自己总结的方法复盘

发表于2026-06-27|更新于2026-06-27|笔记
|总字数:1.4k|阅读时长:4分钟|浏览量:

背景

2026 年 6 月 27 日，花了一个下午搭 Hexo + Butterfly 博客。上线只用了 30 分钟，然后修 bug 修了 4 个小时。

有意思的不是踩了多少坑——而是前几天我刚写完自进化 AI 系统深度对比研究，总结了一堆优秀的设计模式。今天的所有失败，恰恰是把我自己总结的每一条方法论都反着来了一遍。

用那套框架复盘，比单纯”教训 12345”有启发得多。

一、用 Feedback Descent 的视角看操作流程

Feedback Descent 是我从 EvoSkill 源码里提取的核心算法，只有 133 行。它的关键是：

失败时保留反馈，成功时清空历史。

回头看我今天的操作：

每轮失败后没有保留任何反馈。如果每轮结束记录一条：

Feedback Descent 的清空机制也很有意思：只有成功替换才清空历史。 我今天等于是每轮失败都清空了历史——不是算法设计的清空，而是”忘了刚才踩了什么坑”——然后下一轮踩一模一样的。

二、用 dev-log 的视角看 [object Object]

dev-log 是我在分析 self-improving-skills 时发现的一个故事：同一个作者的另一个插件，在 396 个真实 session 里一次都没触发。根因是 grep 了错误的 JSON key。

dev-log 今天
应该检查的 message.content[].name 渲染出的 HTML
实际检查的 "tool":"Edit" 配置文件
结果 396 session 静默失败 4 小时没人发现

根因完全一样：验证的目标和实际输出不一致。 js-yaml.dump() 写出的配置文件看起来对，但模板渲染出来是 [object Object]。我一直在看配置文件，从来没 grep 一下生成出来的 HTML。

analyze_turn.py 里有一个设计我一直记得：“Tool calls are detected via the REAL transcript shape”——用真实的数据结构做判断，不要猜。今天如果改完 YAML 后看了一眼渲染结果，30 秒发现、1 分钟修好。

三、用 skill-distiller 的决策树看操作模式

self-improving-skills 的 skill-distiller 有一个铁律：

我今天全反着来：

  • nanshen CSS 冲突 → 不修优先级，直接新建 custom.css
  • 自定义样式不生效 → 不加 !important 到原文件，而是注入 inject.bottom
  • 配置错误 → 不 git revert，而是 force push 后重来

每一次都是”从零建新方案”而不是”在现有基础上修”。Skill-distiller 的设计思想是保守演进——能 patch 就不新建，能扩展就不重写。今天的操作是激进重来——每次失败都把之前的全推倒。

四、用 Nudge-Once 机制看刹车缺失

analyze_turn.py 里有一个”nudge once per segment”设计：同一个提醒在一个工作段内只弹一次，用户拒绝了就不再重复。

A nudge is raised at most ONCE per segment of work.

今天要有这机制：第一次加 CSS 导致页面崩了，就会被拦住——“上次也是注入 CSS 导致的问题，确认要继续？”

没有刹车，连续犯同样的错。

五、用全局光谱看今天的定位

全局对比 里我把 8 个系统按进化机制的激进程度排了一条光谱：

我今天在模仿最激进的那端（Self-Modify）——直接改线上生产环境。 没有 curator 保留反馈，没有 distiller 沉淀教训，没有 feedback descent 成对比较——就是改→坏→回退→改→坏→回退的随机游走。

如果按保守端来：

  1. Curator：每次失败记录一条 feedback
  2. Distiller：累积 3 条同类 feedback → 蒸馏成 class-level 教训
  3. Feedback Descent：下次遇到同类问题，先对比已知教训和当前方案

这就是 skill-distiller 的设计精神。我分析它的时候觉得”这个决策树很好用”，今天却一次没用上。

六个具体教训

# 教训 对应的框架
1 抄 CSS 前确认版本号 Feedback Descent：成对比较前先验证可比性
2 改之前 git init dev-log 教训：要有降级路径
3 每次只改一件事 skill-distiller：保守演进，能 patch 不重写
4 改完 grep HTML dev-log 教训：验证真实输出，不猜
5 装插件前验证前置 Feedback Descent：先确认 evaluator 可用
6 记录每次失败 Curator：保留反馈才能迭代

总结

不是技术难，是没有用自己总结的方法。 今天之前我已经写了 9 篇分析笔记，把 Feedback Descent、dev-log 教训、skill-distiller 决策树、nudge-once 刹车全部拆清楚写下来了。然后一下午的操作，全部反着来。

最后还用上了一条正确的方法：git init && git tag ok-20260627——这就是 curator 的”保留反馈”。

下次动手之前，先看一眼 ok-20260627 旁边的教训列表。
