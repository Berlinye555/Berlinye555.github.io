花 40 块买来的教训——别什么事都丢给子代理

发表于2026-06-28|更新于2026-06-28|笔记
|总字数:881|阅读时长:2分钟|浏览量:

发生了什么

workspace 里有两个 A 股量化项目，要把老项目的精华搬进新项目。活不大——最后净增 1595 行、4 个新文件、2 个改文件。正常情况下在对话里直接写，一两个小时搞定。

但我手痒了。Claude Code 有个 subagent-driven-development 技能，号称”每个 Task 独立子代理 + 两阶段 review”，看起来很专业。8 个 Task，按这模式走，起了 24 次子代理。

第二天打开 DeepSeek 后台，傻了。

账单说话

正常工作日 (6/27) 子代理实验 (6/28)
总使用量 8,000 万 tokens 6,400 万 tokens
缓存命中 7,900 万 (98.7%) 5,200 万 (81.2%)
费用 ¥4.33 ¥39.61

总使用量其实更少，钱却花了 9 倍。

凶手是缓存命中率

打个比方：外卖的配送费。你点一单 30 块的饭，配送费 3 块；你分 8 单点，每单 4 块的菜，配送费 3×8=24 块。饭钱差不多，配送费炸了。

DeepSeek 的缓存机制就是这样。正常对话是连续追加——前面说过的内容走缓存，只对新内容付费。命中率 99%，便宜。

子代理是 24 次独立 API 调用。每次都换一套 task prompt、换一套代码上下文。缓存对不上，命中率掉到 81%。多出来的 1200 万未命中 tokens 全是钱。

它还不快

有人说子代理能并行、能加速。这台机器的瓶颈是本地跑 Python、读文件、执行命令。子代理的 API 在云端，但工具执行还在地面——每个子代理要重新读代码、重新理解上下文。Review 循环更离谱，同一件事做三遍。

原本一个半小时的活，子代理模式跑了快三个小时。

8 个 Task 逐个鞭尸

Task 干嘛的 该用子代理？
1 git 提交 + 建分支 ❌ 两行命令的事
2 移植形态识别 593 行 ✅ 正经代码
3 创建信号验证 313 行 ✅ 正经代码
4 创建优化引擎 409 行 ✅ 正经代码
5 加 auto_scheme，23 行 ❌ 改个配置文件而已
6 最终验证，跑命令 ❌ 看输出就行
7 删除 test/ 目录 ❌ rm -rf
8 合并到 master ❌ git merge

8 个里 5 个纯属浪费。 git、配置、验证、删除、合并——主对话几秒钟的事，各自起了一个完整子代理，还要配两轮 review。

三个教训

1. “推荐”不等于”适合”

技能模板写着 “Subagent-Driven (recommended)”。那是给 20 个独立模块的大项目准备的。我一个 8 task、5 个琐事的合并计划也照搬，纯属偷懒。

技能是建议，决定权在自己。

2. 每个子代理都计费

一次子代理启动约 ¥0.5-1.5。删个目录花 1 块，改个配置又花 1 块。花之前想想：这个 task 值不值？

3. 缓存是白送的省钱工具

同一个长对话 = 高缓存命中 = 便宜。频繁切上下文 = 低命中 = 贵。不是在教你节省 token，是让你别没事就切上下文。

以后怎么做

一个简单的判断：

  • 主对话 3 分钟能搞定的 → 直接做，不提子代理
  • 净增代码 < 2000 行 → 大概率不需要子代理
  • 只有多文件、复杂逻辑、确实要隔离的 → 再考虑
  • Review 不需要分两拨人 → spec + code 合并成一个就行

¥39.61 买的一课。下次起子代理之前，先问自己：这活值 1 块钱吗？
